The Challenge

Pixel-Perfect Comparison

Our goal is simple yet computationally brutal: Take two temporal satellite captures of the same coordinate, calculate the Mean Squared Error (MSE), and map every change.

The Problem

Big Data Breaks Logic

When images scale to gigabytes, traditional sequential processing doesn't just slow down; it crashes. The CPU starves while waiting for the storage drive (I/O) to feed it pixels.

Sequential Processing Delay

CPU cycles wasted in long I/O wait states.

Parallel Thinking

Breaking the Loaf

To solve this in parallel, we must treat the image like a loaf of bread. We decompose the data into independent workloads for multiple processors.

Strategy 1

Row-wise Slicing

The simplest approach. We slice the image into horizontal strips. Each processor owns a contiguous range of scanlines.

1D Decomposition: High cache efficiency for row-major image formats, as it reads data sequentially from the disk.

Strategy 2

Block-wise Slicing

A checkerboard approach. By dividing into 2D tiles, we reduce the perimeter-to-area ratio, minimizing potential communication overhead in complex spatial algorithms.

2D Decomposition: Ideal for shared-memory grid architectures, but can cause disk thrashing on virtual or mechanical drives.

Strategy 3

The Crown Jewel:
Dynamic Pipeline

Instead of cutting the image into 4 static pieces, we cut it into 400 small "granules." We treat the computation as a continuous task queue.

Task Queue
Logic

Master-Worker Pattern

Master Thread: Manages the queue of all granules.

Worker Threads: As soon as a processor finishes a chunk, it checks out the next task. Faster cores automatically handle more work.

Mechanism

Prefetching

While Processor A is crunching the numbers for Block 10, a background thread is already loading Block 11 into RAM. The application is no longer bound by disk speeds.

Infrastructure

Local vs Cluster

On a local machine, threads fight for a single disk. On an HPC cluster, the Dynamic Pipeline scales beautifully as N processors request tasks simultaneously over high-speed networks.

Benchmark

Normal vs Huge Datasets

Normal: 20-30 MB Image Files. Easily fits entirely in memory.

Huge: 7-8 GB Image Files. Heavy I/O Bottlenecks that force disk thrashing.

Result 1

Small Files: Grid Dominance

Small files fit into cache. Because tasks are short, the "overhead" of the Pipeline (Master talking to Workers) is slower than just letting processors grab a static grid chunk instantly.

Result 2

Huge Files: Pipeline King

Storage drives are sluggish. Reading gigabytes creates massive traffic jams. Scheme 3 chewed through the "Huge" dataset in just 4.86s on 8 cores by hiding read latency via overlapping.

A CPU is a genius that can do anything, one task at a time. A GPU is an army of "simpletons." For satellite imagery, we don't need genius; we need throughput. The GPU handles thousands of pixels simultaneously where the CPU handles dozens.

Why not use AI? Deep Learning is powerful but slow and requires training, but deep learning would be better because it can actually extract features; it has some context, not just blind pixel math. Change Vector Analysis (CVA) is deterministic. It is "Lightweight Math" that runs at the speed of the hardware.

Takeaway 01

Orchestration Over Raw Power

Logic

Parallelism is not just about counting cores. As seen in our MPI implementation, how you decompose data dictates success. Reducing communication overhead and managing downstream flow is the difference between a system that scales and one that chokes.

Takeaway 02

The Art of Fitting In

An engineer's greatest skill is fitting the problem to the resource. We did not let the 8GB limit of our hardware stop us from processing 24GB of data flow. Through Tiling and Streaming, we made the impossible fit into available RAM and VRAM.

Takeaway 03

Physics Always Wins

GPU

You cannot beat physics with clever software alone. Specialized hardware like GPUs will always outperform general-purpose CPUs for data-heavy tasks. By offloading "dumb math" to thousands of tiny cores, we achieved 10x speedups that no CPU optimization could touch.
